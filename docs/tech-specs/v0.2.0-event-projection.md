## 0. Scope and Non-Scope

### In scope (v0.2.0)
- Cloud Postgres tables that receive daemon-ingested events (`daemon_events`)
- Cloud Postgres projection tables (`matches`, `draft_sessions`) that the SPA reads from
- Two new Clerk-protected read endpoints: `GET /api/v1/history/matches` and `GET /api/v1/history/drafts`
- A BFF-internal projection worker (background goroutine) that fans `daemon_events` into the projection tables
- Re-confirming the existing daemon ingest contract (`POST /v1/ingest/events`) so the daemon binary stays compatible

### Out of scope (defer to v0.3.0+)
- Lambda or Step-Functions projection
- Tier enforcement / 30-day window for free users (v0.4.0)
- Cursor-based pagination (v0.2.0 uses page+limit; cursor pagination is a v0.3.0 ergonomic improvement)
- Game-level projection (`games`, `game_plays`, etc.)
- Aggregate stats projection (`player_stats`, `matchup_statistics`)

### Critical clarifications
- **`account_id` type**: The PRD ticket text says `account_id TEXT NOT NULL`. That is **incorrect**. The existing `accounts.id` and `matches.account_id` columns in `internal/storage/migrations/postgres/000054_initial_schema.up.sql` are **`BIGINT`**. This spec uses `BIGINT` to remain consistent. The string the daemon sends (the raw MTGA Arena `account_id` value, e.g., `"3024827a-..."`) is preserved on `daemon_events.account_id` (TEXT) and is mapped to `accounts.id` (BIGINT) by the projection worker via the `accounts.client_id` lookup or by upsert. The DBA ticket must be updated to reflect this — do not introduce a TEXT account_id column on `matches` or `draft_sessions`.
- **Migration directory**: All v0.2.0 cloud-projection migrations land in `services/bff/internal/storage/migrations/postgres/` (the BFF-only migration tree), not the desktop tree at `internal/storage/migrations/postgres/`. The BFF tree already owns `daemon_events` (migration 000061) and `clerk_user_id` (000066). Continue numbering from there.

### 1.2 Migration: extend `daemon_events`

Migration number: **next sequential number after 000066** in `services/bff/internal/storage/migrations/postgres/`. As of 2026-05-06 the next available number is **000067**. Confirm by listing the directory immediately before authoring the file.

**File**: `services/bff/internal/storage/migrations/postgres/000067_add_daemon_events_projection_columns.up.sql`

```sql
-- Add projection cursor and idempotency key to daemon_events.
-- projected_at is set by the projection worker after a row has been written
-- to its destination table (matches / draft_sessions). NULL means "pending".
-- event_id is the daemon-issued unique identifier for the event used for
-- ON CONFLICT idempotency. The daemon already populates this in payload.event_id;
-- promote it to a top-level column so we can index it.

ALTER TABLE daemon_events
    ADD COLUMN IF NOT EXISTS projected_at TIMESTAMPTZ,
    ADD COLUMN IF NOT EXISTS event_id     TEXT;

-- Backfill event_id for any pre-existing rows from payload->>'event_id'.
-- Rows without a payload event_id are left NULL; they will be skipped by the
-- projection worker (logged + projected_at set so they don't re-scan).
UPDATE daemon_events
SET event_id = payload->>'event_id'
WHERE event_id IS NULL;

-- Cursor index: the worker scans WHERE projected_at IS NULL ORDER BY received_at
-- LIMIT 100. Partial index keeps the index tiny because most rows are projected.
CREATE INDEX IF NOT EXISTS idx_daemon_events_pending
    ON daemon_events (received_at)
    WHERE projected_at IS NULL;

-- Idempotency: (user_id, event_id) is unique per daemon. If the daemon retries
-- the same event, the projection ON CONFLICT clauses use this to dedupe.
-- Partial index allows legacy rows with NULL event_id to coexist.
CREATE UNIQUE INDEX IF NOT EXISTS idx_daemon_events_user_event_id
    ON daemon_events (user_id, event_id)
    WHERE event_id IS NOT NULL;
```

**Down migration**: `services/bff/internal/storage/migrations/postgres/000067_add_daemon_events_projection_columns.down.sql`

```sql
DROP INDEX IF EXISTS idx_daemon_events_user_event_id;
DROP INDEX IF EXISTS idx_daemon_events_pending;

ALTER TABLE daemon_events
    DROP COLUMN IF EXISTS event_id,
    DROP COLUMN IF EXISTS projected_at;
```

### 1.3 Final canonical schema (for reference — these tables already exist)

After 000067 lands, the four tables that participate in projection are:

#### `daemon_events` (post-000067)

```sql
CREATE TABLE daemon_events (
    id           BIGSERIAL PRIMARY KEY,
    user_id      BIGINT NOT NULL,                  -- users.id
    account_id   TEXT NOT NULL,                    -- raw MTGA account id from daemon (NOT accounts.id)
    event_type   TEXT NOT NULL,                    -- 'match.completed', 'draft.started', 'draft.pick', etc.
    payload      JSONB NOT NULL,
    occurred_at  TIMESTAMPTZ NOT NULL,             -- when the event happened on the user's machine
    received_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),-- when the BFF received the POST
    event_id     TEXT,                             -- daemon-issued idempotency key (000067)
    projected_at TIMESTAMPTZ                       -- set when worker has projected this row (000067)
);
```

Indexes: `idx_daemon_events_user_occurred` (000061), `idx_daemon_events_pending` (000067), `idx_daemon_events_user_event_id` (000067).

#### `matches` (no change — already exists in 000054)

The relevant columns the projection writes:

```sql
-- Columns written by projection:
id                  TEXT PRIMARY KEY,
account_id          BIGINT NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,

opponent_id         TEXT,

```

Index used by `GET /api/v1/history/matches`: `idx_matches_account_id_timestamp ON matches(account_id, timestamp DESC)` (already created by 000053 / 000054).

#### `draft_sessions` (no change — already exists in 000054)

```sql
-- Columns relevant to projection + history reads:
id              TEXT PRIMARY KEY,
account_id      BIGINT REFERENCES accounts(id) ON DELETE CASCADE,
event_name      TEXT NOT NULL,
set_code        TEXT NOT NULL,
```

## 2. BFF API Contract (backend-engineer — ticket #1396)

### 2.1 Route placement

The desktop BFF (in `internal/api/`) already exposes `POST /api/v1/matches` and `POST /api/v1/drafts` against SQLite. To avoid collision and make the cloud-vs-desktop split unambiguous:

> **Cloud history endpoints live under `/api/v1/history/*`**.

Concretely:
- `GET /api/v1/history/matches` — Postgres-backed, Clerk-scoped match history
- `GET /api/v1/history/drafts` — Postgres-backed, Clerk-scoped draft history

These routes must be mounted inside the existing Clerk-protected route group in `services/bff/cmd/main.go` (the `r.Group(func(r chi.Router) { r.Use(deps.ClerkAuthMiddl); r.Use(deps.ClerkUserResolver); ... })` block — see lines 240-261 of main.go).

`UserIDFromContext(ctx)` returns the int64 `users.id`. The handler must then resolve the set of `accounts.id` rows owned by that user (via `accounts.user_id = $1`) to scope the query. For v0.2.0, assume one account per user — the handler may use `SELECT id FROM accounts WHERE user_id = $1 LIMIT 1` and pass that single account_id to the matches/drafts query. Multi-account fan-out is v0.3.0.

### 2.2 GET /api/v1/history/matches

**Auth**: `Authorization: Bearer <Clerk JWT>`. ClerkAuthMiddleware enforces; ClerkUserResolver populates `UserIDFromContext`.

**Query parameters**:

| Name | Type | Default | Constraints | Description |
|---|---|---|---|---|
| `page` | int | `1` | `>= 1` | 1-indexed page number |
| `limit` | int | `20` | `1 <= limit <= 100` | Page size; values outside the range return `400` |
| `format` | string | unset | matches a known MTGA format string (case-insensitive); unknown values return `400` | Optional filter: `Standard`, `Historic`, `Brawl`, `Limited`, `Draft`, `Sealed`, etc. |

**Response shape (200)**:

```json
{
  "data": [
    {
      "id": "match_01HXY...",
      "format": "Standard",
      "result": "win",
      "timestamp": "2026-05-05T18:42:11Z",
      "duration_seconds": 612,
      "deck_id": "deck_01HXY...",
      "rank_before": "Gold-2",
      "rank_after": "Gold-1",
      "opponent_rank": "Gold-1",
      "player_wins": 2,
      "opponent_wins": 1
    }
  ],
  "total": 137,
  "page": 1,
  "limit": 20
}
```

Field notes:
- `id`: `matches.id` (TEXT)
- `format`: `matches.format` (TEXT)
- `result`: `matches.result` (one of `"win"`, `"loss"`)
- `timestamp`: `matches.timestamp` serialized as RFC 3339 / ISO 8601 UTC
- `duration_seconds`: nullable int (omit field or send `null` if not recorded)
- `deck_id`: nullable string (omit or `null`)
- `rank_before` / `rank_after`: nullable strings
- `opponent_rank`: derived in v0.2.0 — return `null` if unknown. (The desktop schema does not store opponent rank as a first-class column; if the projection cannot derive it from the daemon payload, return `null` and revisit in v0.3.0.)
- `player_wins` / `opponent_wins`: ints
- Empty result: return HTTP 200 with `"data": []` and the actual `total` (which will be `0`). **Never return 404 for an empty list.**

Ordering: `ORDER BY timestamp DESC`.

**Errors**:
- `401 Unauthorized` — missing or invalid Clerk JWT (handled by middleware; body: `{"error":"unauthorized"}`)
- `400 Bad Request` — invalid query param (`page < 1`, `limit > 100`, unknown `format`); body: `{"error":"<specific reason>"}`
- `500 Internal Server Error` — DB error; body: `{"error":"internal server error"}`; full error logged to Sentry

### 2.3 GET /api/v1/history/drafts

**Auth**: same as 2.2.

**Query parameters**:

| Name | Type | Default | Constraints | Description |
|---|---|---|---|---|
| `page` | int | `1` | `>= 1` | 1-indexed page number |
| `limit` | int | `20` | `1 <= limit <= 100` | Page size |
| `set_code` | string | unset | 3-5 uppercase letters; otherwise `400` | Filter by MTGA set (`EOE`, `BRO`, `MOM`, etc.) |

**Response shape (200)**:

```json
{
  "data": [
    {
      "id": "draft_01HXY...",
      "set_code": "EOE",
      "format": "PremierDraft",
      "started_at": "2026-05-04T22:10:01Z",
      "completed_at": "2026-05-04T22:48:33Z",
      "wins": 5,
      "losses": 1
    }
  ],
  "total": 12,
  "page": 1,
  "limit": 20
}
```

Field notes:
- `id`: `draft_sessions.id` (TEXT)
- `set_code`: `draft_sessions.set_code` (TEXT)
- `format`: derived from `draft_sessions.draft_type` (e.g., `quick_draft` → `"QuickDraft"`, `premier_draft` → `"PremierDraft"`); use draft_type verbatim if a clean mapping is not obvious
- `started_at`: `draft_sessions.start_time` (RFC 3339)
- `completed_at`: nullable; `draft_sessions.end_time` (RFC 3339) — `null` if draft is still in progress
- `wins`, `losses`: aggregate counts via `LEFT JOIN draft_match_results dmr ON dmr.session_id = draft_sessions.id GROUP BY draft_sessions.id` summing `result = 'win'` and `result = 'loss'`. Default to `0` if no match results yet.
- Empty result: HTTP 200, `"data": []`, `"total": 0`. **Never 404.**

Ordering: `ORDER BY start_time DESC` (`draft_sessions.start_time`, not `created_at`, to match user expectations).

**Errors**: identical to 2.2 (401 / 400 / 500).

### 2.4 Pagination contract details

- `total` is the **unfiltered-by-pagination** count of rows matching the auth+filter scope. Compute with a separate `SELECT COUNT(*)` query — for v0.2.0 the row counts per user are small enough (low thousands max) that the cost is acceptable.
- `page` and `limit` are echoed back in the response for client convenience.
- `OFFSET` is computed as `(page - 1) * limit`.
- Out-of-range `page` (e.g., `page=99` when only 2 pages exist) returns HTTP 200 with `"data": []` and the correct `total`. Do not 404.
- Cursor pagination is explicitly **not** in v0.2.0. Do not add a `cursor` query param. The PRD ticket text mentions cursor — that text is incorrect; this spec overrides.

### 2.5 Acceptance for the BFF endpoint ticket (#1396)

- [ ] Both routes mounted inside the `r.Group` Clerk-protected block in `services/bff/cmd/main.go`
- [ ] Handlers in `services/bff/internal/api/handlers/history.go` (new file)
- [ ] Repository methods in `services/bff/internal/storage/repository/matches_repo.go` and `draft_sessions_repo.go` (new files) — every query parameterised by `account_id`
- [ ] Integration tests cover: 200 happy path, 200 empty, 401 missing JWT, 400 invalid `limit`, 400 invalid `format`, 400 invalid `set_code`, pagination correctness (page 2 returns the next 20 rows)
- [ ] Cross-tenant scoping test: a request authenticated as user A never returns user B's rows (this is a critical multi-tenant correctness test)
- [ ] No N+1 queries in the drafts endpoint — wins/losses computed in a single GROUP BY join
- [ ] Architect pre-push review approved before push

## 3. Projection Worker Design (backend-engineer — ticket #1396)

### 3.1 Location and lifecycle

- **File**: `services/bff/internal/projection/worker.go`
- **Started from**: `services/bff/cmd/main.go`, after DB and repos are wired but before the HTTP server listens. Use a `context.Context` derived from a top-level cancel so SIGTERM during graceful shutdown stops the loop cleanly.
- **Concurrency**: single goroutine (no fan-out). One worker is sufficient for v0.2.0 throughput (one user, low event rate).
- **Trigger**:
  1. Run once on startup (drains any backlog).
  2. Then loop with a `time.NewTicker(30 * time.Second)`. Each tick calls `runOnce(ctx)`.
- **Disable switch**: env var `BFF_PROJECTION_DISABLED=true` skips starting the worker. This lets ops disable projection without redeploying if a bad event causes a hot loop.

### 3.2 Per-tick query

```sql
SELECT id, user_id, account_id, event_type, payload, occurred_at, received_at, event_id
FROM daemon_events
WHERE projected_at IS NULL
ORDER BY received_at ASC
LIMIT 100;
```

The partial index `idx_daemon_events_pending` makes this an index-only scan over pending rows. Process each row sequentially.

### 3.3 Per-row processing

Switch on `event_type`:

| `event_type` | Destination | Action |
|---|---|---|
| `match.completed` | `matches` | Decode payload into a Match struct; resolve `accounts.id` from `accounts.client_id = daemon_events.account_id` (insert into `accounts` if missing); UPSERT into `matches` |
| `draft.started` | `draft_sessions` | UPSERT (insert if `id` is new; update `start_time` if already exists) |
| `draft.pick` | `draft_sessions` (update `total_picks`) and optionally `draft_picks` | UPSERT pick row; bump `draft_sessions.total_picks`, `updated_at` |
| `draft.completed` | `draft_sessions` | Set `end_time`, `status='completed'` |
| any other | (skip) | Set `projected_at = NOW()` so we don't re-scan; emit a single INFO log line per unknown type per worker run |

After successful per-row processing (or skip), update the cursor:

```sql
UPDATE daemon_events SET projected_at = NOW() WHERE id = $1;
```

This UPDATE happens in the **same DB transaction** as the projection write so partial failures don't strand a row.

### 3.4 Idempotency contract

For `matches`:

```sql
INSERT INTO matches (id, account_id, event_id, event_name, timestamp, ...)
VALUES (...)
ON CONFLICT (id) DO UPDATE
SET event_name = EXCLUDED.event_name,
    timestamp  = EXCLUDED.timestamp,
    ...
    -- never overwrite created_at
```

For `draft_sessions`:

```sql
INSERT INTO draft_sessions (id, account_id, event_name, set_code, draft_type, start_time, status, ...)
VALUES (...)
ON CONFLICT (id) DO UPDATE
SET end_time    = COALESCE(EXCLUDED.end_time, draft_sessions.end_time),
    total_picks = GREATEST(EXCLUDED.total_picks, draft_sessions.total_picks),
    status      = EXCLUDED.status,
    updated_at  = NOW();
```

The `id` PK on each destination table provides natural idempotency. Re-projecting the same `daemon_events` row produces the same destination row; running the worker twice on the same backlog produces no duplicates.

### 3.5 Error handling — malformed payload policy

**Decision**: malformed payloads are **logged and marked projected**, not retried. Rationale:
- A malformed payload is a permanent failure (the daemon shipped bad data). Retrying it forever would create a hot loop on a poison row.
- Marking `projected_at = NOW()` removes the row from the worker's scan window so it stops blocking the queue.
- The raw row stays in `daemon_events` indefinitely so an operator can inspect and reprocess manually if needed.
- The error is reported to Sentry with `event_type`, `daemon_events.id`, and the full payload truncated to 4 KB.

Do **not** crash the worker on a malformed row. The defer-recover at the top of `runOnce` catches panics, logs them, and continues.

### 3.6 Observability

For each `runOnce` invocation, emit one structured log line at the end:

```
[projection] runOnce completed pending=N projected=K skipped_unknown=U skipped_malformed=M errored=E duration_ms=X
```

Counters are also exported as Prometheus metrics if the metrics endpoint is wired (out of scope for this spec — log lines are the v0.2.0 floor).

## 4. Daemon Ingest Endpoint (reference — already implemented)

This section documents the existing contract so engineers do not re-design it. **Do not change this contract in v0.2.0.** Daemon binary compatibility is a hard constraint.

### 4.1 Route

```
POST /v1/ingest/events
```

Mounted in `services/bff/cmd/main.go` line 218-225. Note the `/v1/` prefix (no `/api/`) — this is intentional and is the daemon's stable contract.

### 4.2 Auth

`APIKeyAuth` middleware. The daemon sends:

```
Authorization: Bearer <api_key>
```

where `<api_key>` is a key generated for the user via `POST /api/keys` and stored in the daemon's local config. **Not Clerk JWT.** Clerk is for browser sessions only; the daemon is a long-lived headless binary and uses API keys.

The legacy `DAEMON_JWT_SECRET` HMAC path was removed (#1315 / ADR-009) and must not be reintroduced.

### 4.3 Request body

The daemon sends a single `contract.DaemonEvent` per POST (see `services/contract/contract.go`):

```json
{
  "type": "match.completed",
  "account_id": "3024827a-1234-5678-9abc-def012345678",
  "event_id": "evt_01HXY...",
  "occurred_at": "2026-05-05T18:42:11Z",
  "payload": { /* event-type-specific */ }
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `type` | string | yes | `match.completed`, `draft.started`, `draft.pick`, `draft.completed` |
| `account_id` | string | yes | Raw MTGA Arena account id (NOT the `accounts.id` BIGINT — this is the user-facing account string) |
| `event_id` | string | yes | Daemon-generated UUID/ULID for idempotency (post-000067) |
| `occurred_at` | RFC 3339 string | yes | When the event happened on the user's machine (not when received) |
| `payload` | JSON object | yes | Event-type-specific; preserved verbatim into `daemon_events.payload` |

The handler decodes the event, persists it via `DaemonEventsRepository.Insert`, and broadcasts it via SSE. The persistence call needs to be updated to also write `event_id` once migration 000067 lands — the backend-engineer must include this in the same PR as the migration consumer.

### 4.4 Response

- `202 Accepted` — event persisted (or attempted) and broadcast. Empty body.
- `400 Bad Request` — JSON decode failure or missing `type`.
- `401 Unauthorized` — missing/invalid API key.

The handler **always returns 202 even if persistence failed** (it logs the error and broadcasts anyway). This is intentional: the SSE consumer should still see live events when the DB is degraded. Do not change this behavior in v0.2.0.

### 4.5 What the backend-engineer must update in v0.2.0

The current `IngestHandler` does not pass `event_id` to the repository because the column does not exist yet. After 000067 lands:

1. Update `DaemonEventInserter.Insert` signature to accept `eventID string`
2. Update `DaemonEventsRepository.Insert` to write the `event_id` column
3. Update the handler to pass `event.EventID` from the contract struct
4. Add `EventID string \`json:"event_id"\`` to `contract.DaemonEvent` if not already present (verify in `services/contract/contract.go`); if added, tag and bump the contract module version per the Go workspace rules

This change set is part of ticket #1396, not a separate ticket.

## 5. Cross-cutting constraints

### 5.1 Multi-tenancy (ADR-009)

Every query against `matches` or `draft_sessions` must include `account_id = $X` in the WHERE clause. The repository layer is responsible for this; handlers must never construct ad-hoc queries that bypass scoping.

A repository unit test must demonstrate that a query for user A's data returns zero rows when only user B's data exists.

### 5.2 Sentry

Per the v0.2.0 PRD, BFF errors must be reported to Sentry. The projection worker's malformed-payload path explicitly reports to Sentry with the daemon_events.id and payload (truncated). DB errors in the history endpoint handlers also report to Sentry before returning 500.

### 5.3 Forbidden patterns (per CLAUDE.md)

- No custom JWT verification — `clerk-sdk-go v2` only on the BFF
- `DAEMON_JWT_SECRET` is M2M only (and being removed in #1315) — no user-facing auth uses it
- No `// TODO: add auth later` placeholders on protected routes
- Do not duplicate Clerk session state on the frontend; the SPA wiring (out of scope for this spec) uses `useAuth()` / `useUser()` only

### 5.4 Tests required

Per CLAUDE.md test coverage guidelines:
- Repository integration tests against a real Postgres (testcontainers or shared CI Postgres) — not mocks
- Handler-level HTTP tests for both endpoints, including the cross-tenant correctness case
- Projection worker integration test driving real rows through the full flow

## 7. Open questions deferred to v0.3.0

- **Multi-account fan-out**: when a user has multiple MTGA accounts, the history endpoints will need to either accept an `account_id` query param or aggregate across all accounts. v0.2.0 assumes one account per user.
- **Opponent rank**: the desktop schema does not record this. If the daemon payload contains it, the projection should populate a new column; if not, it stays `null` and we revisit.
- **Game-level projection**: the cloud projection only writes `matches` and `draft_sessions` in v0.2.0. Game-by-game data (`games`, `game_plays`) waits for v0.3.0 demand.
- **Cursor pagination**: page+limit is sufficient for low row counts. When per-user row counts grow past ~10k we revisit.
