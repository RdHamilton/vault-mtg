# ADR-020: Daemon Initial Auth Acquisition: PKCE Browser Redirect

| Field       | Value                          |
|-------------|-------------------------------|
| **Status**  | Accepted                       |
| **Date**    | 2026-05-09                     |
| **Author**  | backend-engineer               |
| **Tickets** | #1643, #1650, #1651, #1652, #1674 |

---

## Context

The daemon needs a user-scoped API key to call `POST /v1/ingest/events`. Previously
this key was provisioned manually (copy-paste from the SPA) and stored in plaintext
inside `daemon.json`. This approach is error-prone, breaks first-run UX, and stores
credentials insecurely on disk.

Wave 3 replaces the manual flow with a zero-terminal PKCE browser-redirect flow:
the daemon detects that no valid config exists, opens the system browser to
`vaultmtg.app/setup` (or prints the URL if headless), completes Clerk PKCE OAuth,
exchanges the auth code for a session token, calls the BFF to mint an API key,
and stores that key in the OS keychain ŌĆö never in a plaintext file.

---

## Decision

### Flow (step by step)

1. **First-run detection** ŌĆö daemon starts, finds no `daemon.json` (or
   finds a stub with no `api_key` and no keychain sentinel).

2. **PKCE setup** ŌĆö daemon generates a cryptographically random
   `code_verifier` (32 bytes, base64url) and derives `code_challenge`
   (SHA-256 of verifier, base64url).

3. **Localhost callback server** ŌĆö daemon binds a one-shot HTTP server on
   **fixed port `51423`** (one retry on `51424` if busy). The redirect URI is
   `http://localhost:51423/oauth/callback` (or `51424` on retry).
   Clerk OAuth redirect URIs **must be registered for both**:
   - `http://localhost:51423/oauth/callback`
   - `http://localhost:51424/oauth/callback`

4. **Browser open** ŌĆö daemon constructs the Clerk OAuth authorization URL
   with `response_type=code`, `code_challenge`, `code_challenge_method=S256`,
   and `redirect_uri`. Daemon calls the OS "open URL" command
   (`open` on macOS, `start` on Windows) to launch the system browser.

5. **User authenticates** ŌĆö user logs in to their Clerk account in the
   browser. Clerk redirects to `localhost:PORT/oauth/callback?code=AUTH_CODE`.

6. **Code capture** ŌĆö daemon's callback server receives the request,
   extracts `code`, shuts down the listener.

7. **Token exchange** ŌĆö daemon POSTs `code` + `code_verifier` to Clerk's
   token endpoint. Clerk returns a session token (JWT).

8. **Daemon registration** ŌĆö daemon calls `POST /v1/daemon/register` on the
   BFF with the Clerk JWT as `Authorization: Bearer`. BFF verifies the JWT
   via `clerk-sdk-go v2`, creates (or retrieves) a per-account API key
   scoped to the authenticated user's `account_id`, and returns the key in
   the response body. See ┬¦ "POST /v1/daemon/register Wire Format" below.

9. **Keychain storage** ŌĆö daemon stores the API key in the OS keychain
   using `go-keyring` (service name: `com.mtga-companion.daemon`, account: `api-key`).
   The key is NOT written to `daemon.json` in plaintext.
   `com.mtga-companion.daemon` is the canonical service name.

10. **Config write** ŌĆö daemon writes `daemon.json` with `cloud_api_url`
    and `"keychain": true` (flag indicating the API key lives in the keychain,
    not the config file). All subsequent restarts read from the keychain.

### Headless / CI fallback

When `open` / `start` fails (headless server, no display), the daemon logs:

```
[pkce] browser could not be opened ŌĆö complete auth at:
  http://localhost:51423/oauth/callback
  Clerk auth URL: https://clerk.vaultmtg.com/oauth/authorize?...
```

The daemon still binds the callback port and waits up to 5 minutes for a
redirect. This lets power users copy-paste the URL into a local browser.

## daemon.json Canonical Schema

### v2 schema (post-Wave 3)

```json
{
  "cloud_api_url": "https://api.vaultmtg.app",
  "keychain": true,
  "sync_enabled": true,
  "account_id": "user_2abc...",
  "log_path": "",
  "ingest_path": "/v1/ingest/events"
}
```

| Field         | Type    | Required | Notes                                              |
|---------------|---------|----------|----------------------------------------------------|
| `cloud_api_url` | string  | yes      | Base URL of BFF; no trailing slash                |
| `keychain`    | bool    | no       | `true` = API key lives in OS keychain             |
| `sync_enabled`| bool    | no       | Default `true`; set `false` to disable event dispatch |
| `account_id`  | string  | no       | Clerk user ID; used to tag ingest events          |
| `log_path`    | string  | no       | Override MTGA log path; auto-detected if empty    |
| `ingest_path` | string  | no       | Default `/v1/ingest/events`                       |

**Fields NOT present in v2 schema:**
- `api_key` ŌĆö moved to OS keychain; any existing plaintext value is ignored on startup.

### Migration path from legacy plaintext `api_key`

On startup the daemon follows this sequence:

1. Load `daemon.json` (if it exists).
2. If `keychain: true` ŌĆö read API key from OS keychain. Done.
3. If `api_key` is non-empty (legacy plaintext) ŌĆö migrate:
   a. Store `api_key` value into OS keychain.
   b. Delete `api_key` from in-memory config.
   c. Set `keychain: true`.
   d. Write updated `daemon.json` (without `api_key`, with `keychain: true`).
4. If neither `keychain` nor `api_key` ŌĆö first-run detected; start PKCE flow.

This migration is transparent: users who already configured the daemon with a
plaintext key are upgraded silently on the next daemon start.

## POST /v1/daemon/register Wire Format

### Request

```
POST /v1/daemon/register
Authorization: Bearer <clerk_session_jwt>
Content-Type: application/json

{}
```

The request body is intentionally empty. All user identity is derived from
the verified Clerk JWT (`sub` claim = Clerk user ID).

### Response ŌĆö 200 OK (key already exists)

```json
{
  "api_key":    "mtga_sk_<64-hex-chars>",
  "account_id": "user_2abc..."
}
```

Returns `200` when the account already has an API key. The existing key is
returned (not rotated). This enables reinstall without re-pairing.

### Response ŌĆö 201 Created (new key minted)

```json
{
  "api_key":    "mtga_sk_<64-hex-chars>",
  "account_id": "user_2abc..."
}
```

Returns `201` when a new key is minted for this account.

### Error responses

| Status | Condition                                        |
|--------|--------------------------------------------------|
| 401    | Missing, expired, or invalid Clerk JWT          |
| 429    | Rate limit exceeded (5 req/min per account_id)  |
| 500    | Internal server error (key generation/storage)  |

### Key format

```
sk_live_<64 hex characters (32 random bytes, hex-encoded)>
```

The literal prefix `sk_live_` is followed by 64 lowercase hex characters.
Example representation: `sk_live_` + 64 hex chars = 72 characters total.

The plaintext key is shown **exactly once** in the response and is never stored
on the BFF ŌĆö only a bcrypt hash is persisted in `daemon_api_keys`.

## daemon_api_keys Table Schema

```sql
CREATE TABLE daemon_api_keys (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id  TEXT NOT NULL,
  key_hash    TEXT NOT NULL,
  key_prefix  TEXT NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_used   TIMESTAMPTZ,
  revoked_at  TIMESTAMPTZ,
  UNIQUE (account_id)
);
```

- `account_id` ŌĆö Clerk user ID (string, e.g. `user_2abc...`)
- `key_hash` ŌĆö bcrypt hash of the plaintext API key
- `key_prefix` ŌĆö first 16 chars of plaintext for display/debugging only
- `UNIQUE (account_id)` ŌĆö one key per user for beta

## Keychain Storage

| Property       | Value                          |
|----------------|-------------------------------|
| Service name   | `com.mtga-companion.daemon`   |
| Account name   | `api-key`                     |
| Library        | `go-keyring` (CGO-free)       |
| macOS backend  | macOS Keychain                |
| Windows backend| Windows Credential Manager    |

The `go-keyring` library is CGO-free and cross-compiles cleanly for
`GOOS=windows GOARCH=amd64` without any native toolchain on the build machine.

---

## PKCE Callback Ports

| Port  | Role                            |
|-------|---------------------------------|
| 51423 | Primary callback port           |
| 51424 | Fallback (one retry if 51423 busy) |

Both `http://localhost:51423/oauth/callback` and
`http://localhost:51424/oauth/callback` are registered as allowed redirect URIs
in Clerk Native Applications.

---

## Consequences

**Positive:**
- Zero manual key copy-paste for first-run daemon setup.
- API key never stored in plaintext on disk.
- Idempotent re-install: same key returned on re-pair.
- CGO-free build constraint satisfied.

**Negative / tradeoffs:**
- Requires system browser. Headless installs need a copy-paste fallback (documented above).
- Fixed ports 51423/51424 must not be in use. Probability is low; error message is actionable.

**Out of scope:**
- Key rotation UI (post-GA).
- Multi-device pairing (post-GA; current constraint is one key per account).
- Custom URL scheme (`vaultmtg://callback`) ŌĆö not needed given registered ports.
