## Executive Summary

After synthesizing BA market research, Growth Marketing analysis, and founder vision clarification (2026-05-08), v0.4.0 is restructured around three strategic priorities:

1. **Core analytics** (#1541, #1540) — free constructed win rate dashboard and Brawl commander analytics ship in Wave 1. These require no new infrastructure beyond v0.3.0 telemetry, serve the top two Arena formats, and directly challenge Untapped.gg's paid analytics wall.

2. **ML deck building — collection-aware, personalized** — the ML engine is repositioned, not sidelined. The correct vision (per Ray) is not "draft pick recommendations competing with 17Lands" — it is "given what I own right now, what is the best deck I can build, and what should I craft next?" This is a new product category with no direct competitor. It ships in Wave 2 in degraded mode (full card pool, no personal collection required) while collection log parsing is investigated in parallel.

3. **Shareable profile** (#1542) — deferred to Wave 3 because it depends on Wave 1 data being stable and populated. A shareable profile with empty stats harms first impressions.

The prior framing of the ML engine as a 17Lands competitor is explicitly abandoned. Draft monitoring remains central — every draft session is a guaranteed ML application context where the card pool is constrained and the goal is clear (build the best deck from these picks). The distinction: VaultMTG does not tell players which card has the highest aggregate win rate (17Lands). VaultMTG tells players which card best completes the deck they are building from their current draft pool and existing collection.

## ML Vision — Corrected Framing

The ML feature is NOT draft pick recommendations. The correct vision is:

> **"Given what I own right now, what is the best deck I can build — and what should I craft next?"**

This is **collection-aware, personalized deck building ML** targeted at casual and F2P players who play infrequently and need to make every wildcard count.

### Why this is a new category, not a feature arms race

| Competitor | What they do | What they miss |
|---|---|---|
| 17Lands | Draft pick win-rate recommendations | No collection awareness; no constructed optimization |
| Untapped.gg | Meta decks, opponent archetype tracking | Has a deck builder but no real-time collection-gated recommendations; Pro paywall for most useful analytics |
| Moxfield | Deck theory and archetype browsing | No real-time collection awareness; no personalized "what to craft next" path |

No major competitor delivers: "here is the best deck you can build right now from the cards you actually own, and here is the one wildcard that unlocks the most competitive upgrade." This is the VaultMTG differentiator.

### Two delivery modes

| Mode | Collection data required | Recommendation basis | When |
|---|---|---|---|
| Degraded | No | Full card pool — assumes access to all cards. Useful for format orientation and wildcard budget planning. | Wave 2 |
| Full | Yes — log parsing confirmed reliable | Personal collection — personalized gap analysis, exact wildcard priority, "best deck right now" | Wave 3 (conditional) |

The degraded mode has standalone value: players without a tracked collection get format-relevant deck suggestions from competitive archetypes ranked by meta win rate. The full mode is the differentiated product.

### Collection data — critical infrastructure dependency

Collection data is the prerequisite for the full ML value proposition. Without it, "what to craft next" is generic, not personalized. Two sourcing paths exist:

- **Log parsing (primary)**: The MTGA Arena client log contains collection state. A time-boxed spike in Wave 2 validates whether the daemon can reliably extract card ownership and wildcard counts from these logs.
- **Overwolf (fallback)**: The Overwolf GEP provides a dedicated collection API for Arena companions. This path requires joining the Overwolf developer program and shipping within Overwolf's overlay SDK. Significant architecture cost and Windows-only lock-in. Scoped only if log parsing is ruled out — post-GA consideration.

The log parsing spike runs concurrently with Wave 2 ML degraded mode development. It does not block Wave 2 from starting.

## Open Questions — Ray's Decision Required Before Kickoff

The following questions must be answered before any v0.4.0 feature tickets move to In Progress. Engineering should not start Wave 1 without these resolved.

1. **ML degraded mode first**: Should the ML deck building feature ship in degraded mode (full card pool, no personal collection) to get something in users' hands immediately, while collection data is being solved in parallel during Wave 2? The current proposal says yes — degraded mode ships in Wave 2 as a standalone value-add, and full personalized mode follows in Wave 3 if the collection spike returns GO. Confirm this is the right sequencing or redirect.

2. **Username system for shareable profiles (#1542)**: Shareable profiles at `vaultmtg.app/u/[username]` require users to have a public handle. Clerk provides `userId` but not a user-chosen username. Options: (a) use a display name field added to user profile, (b) auto-generate a handle from Clerk email prefix, (c) let users set a custom handle stored in VaultMTG's own DB. Which approach? Note: if we need a custom handle, that is an extra DB table + UI that adds engineering scope to Wave 3.

3. **Opponent archetype breakdown scope (#1541)**: "Opponent archetype" analytics require either (a) user-tagged opponent decks or (b) automated archetype detection from the Arena log (opponent cards seen). How much opponent card visibility does the log expose? If the log does not reliably show opponent deck composition, the opponent archetype feature may need to be descoped to "win rate by format + deck only" for Wave 1. BA spike (3 days) would answer this — should this run before or in parallel with Wave 1 development?

4. **Collection investigation trigger (#1543)**: The spike will produce a go/no-go. If it is a no-go, the ML full mode (collection-aware personalization) is parked pending Overwolf evaluation post-GA, and Wave 3 ships only shareable profile and ML degraded mode refinement. Are you comfortable with that outcome, or is collection tracking a must-ship for v0.4.0 regardless of log feasibility?

5. **Clerk Pro decision (#1398, #1314)**: This was deferred to 2026-05-09. Confirm whether Clerk Pro is approved and whether #1398 (daemon onboarding) and #1314 (API key UX) move forward in parallel with Wave 1 or remain parked.

6. **v0.4.0 beta target date**: The BA synthesis recommends shipping Wave 2 before SCG CON Dallas (September 4–6, 2026) for event marketing value. Is July 7, 2026 still the Wave 1 target, or has v0.3.0 slippage shifted this? Confirm the date constraint before Wave 1 tickets are sized.

## What Was Cut and Why

| Cut item | Reason |
|---|---|
| Draft ML — win-rate pick advisor (17Lands-style) | 17Lands has years of aggregate training data VaultMTG cannot match at pre-50K MAU scale. A model trained on a small user base produces worse win-rate picks than 17Lands, not better. Shipping an inferior pick advisor would harm brand perception. Note: deck *synthesis* ML (which pick completes your archetype) is IN — this cut is specifically the 17Lands-style aggregate win-rate advisor. |
| 17Lands data integration | No standalone value without the win-rate pick model. The deck synthesis ML uses the daemon's own draft log data, not 17Lands aggregate rates. |
| ML infrastructure / retraining pipeline | No model means no pipeline. Infrastructure cost ($0 now but non-trivial ongoing if built) is not justified by beta-scale usage. |
| Full standalone deck builder | Moxfield has 10+ years of UX polish and a massive community. Building a competing deck builder is not wrong — it is wrong at this stage. The collection export path (Moxfield integration) captures most of the user value at a fraction of the cost. Revisit as a post-GA initiative if collection export adoption is high. |
| Alchemy features | ~8–12% format share and declining. Divisive among players. Engineering investment delivers limited acquisition value. |

**Scoring notes**:
- Win rate dashboard wins Wave 1 priority by both score and confidence. Broad format coverage makes it the highest-reach feature in v0.4.0.
- Brawl analytics scores below win rate dashboard but ships in Wave 1 because it shares the same match data pipeline — marginal engineering cost to run in parallel.
- ML degraded mode confidence is 75%: the recommendation engine approach is well-understood; risk is mainly in surfacing results meaningfully in UI.
- ML full mode confidence drops to 55% because it is gated on collection spike GO and the quality of log-parsed collection data.
- Shareable profile scores high but moves to Wave 3 — not a confidence issue, a data quality dependency. A profile with empty stats shipped before Wave 1 data matures would harm the feature's reception.
- Draft ML win-rate advisor (sidelined) is not a close call: effort/confidence combination produces a score 75x worse than the win rate dashboard. Draft ML deck synthesis (IN) scores competitively at 2,100 because it uses existing daemon log data with no external dependency.
