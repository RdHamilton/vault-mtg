## 1. Purpose

v0.3.0 is the internal beta gate. When this milestone closes, 10–20 trusted testers are invited to validate the six CS beta gates. No external users before v0.4.0.

The work divides into two parallel tracks:

- **Telemetry track**: parse all Arena log event types in the daemon, project them into the full domain schema, expose read endpoints.
- **Live draft track**: SSE consumer, `/draft/live` page, live card grades during a draft session.

Both tracks must complete for the milestone to close. The live draft track (tickets #1387–#1393) was created earlier; this document integrates it into the full wave plan.

## 4. User Stories with Acceptance Criteria

### Wave 0 — Spike Verification

**As an engineer starting v0.3.0,**
**I want** confirmation that the existing parsers in `internal/mtga/logreader/` correctly parse current Arena log output,
**so that** the extraction and classifier work does not start on a broken foundation.

**ACs:**
- [ ] Given a current Arena session (not older than 7 days), when 3 representative log samples are captured (match game, draft, collection update), then each sample can be parsed by the existing parser without errors
- [ ] Given the parsed output, when it is diffed against the existing testdata files, then any structural drift is documented in a comment on #1501
- [ ] Given the spike result, a go/no-go signal is posted on #1501 before any classifier ticket is started

---

### Wave 1 — Parser Extraction

**As a backend engineer working on BFF projectors,**
**I want** the parser logic available as a shared `pkg/logparse` package,
**so that** I can import parsing primitives without duplicating code.

**ACs:**
- [ ] Given `pkg/logparse` is created, when imported in both the daemon and the BFF, then both compile with no import cycle violations
- [ ] Given all existing parser unit tests, when migrated to `pkg/logparse`, then they all pass
- [ ] Given the desktop binary, when `internal/mtga/logreader` imports are replaced with `pkg/logparse`, then `go build ./...` for the desktop module succeeds
- [ ] Given the CI pipeline, when the extraction PR merges, then all three binaries (daemon, BFF, desktop) are compiled in the same run

**As a DBA,**
**I want** all telemetry tables to have `account_id` scoping and the required new columns,
**so that** the projection layer can write multi-user-safe data.

**ACs:**
- [ ] Given 5 migrations (account_id on quests, quest_session_tracking, inventory_tracking, game_plays, life_change_tracking), when applied, then existing rows are backfilled and no NOT NULL violations exist
- [ ] Given the `game_plays` table, when the `partial BOOLEAN NOT NULL DEFAULT FALSE` migration runs, then the column exists with the correct default
- [ ] Given the `daemon_events` table, when the `sequence BIGINT NOT NULL DEFAULT 0` migration runs, then the column exists with an index on `(account_id, sequence)`
- [ ] Given all migrations, when applied and rolled back, then the database returns to its prior state (reversible)

---

### Wave 2 — Daemon Classifiers

**As a VaultMTG user with the daemon running,**
**I want** the daemon to capture inventory, quest, collection, deck, and match game events from my Arena session,
**so that** my complete game activity is visible in the app.

**ACs (per classifier):**
- [ ] Given a log entry matching the target event type, when the daemon processes it, then a correctly-typed event is dispatched to the BFF ingest endpoint
- [ ] Given a log entry that does not match, when the classifier runs, then no event is emitted (miss case)
- [ ] Given testdata samples from #1501, when unit tests run, then all classifier match/miss/payload cases pass
- [ ] Given the daemon is running and a game ends (no `game_end` log entry — e.g., Arena crash), when the stale-buffer sweep runs after 15 minutes, then a partial `GamePlayEvent` is dispatched and the buffer is cleared

---

### Wave 3 — BFF Projectors

**As a VaultMTG user whose daemon has sent events,**
**I want** those events projected into structured domain rows,
**so that** analytics endpoints have data to query.

**ACs:**
- [ ] Given a dispatched `inventory.updated` event, when the inventory projector runs, then a row is written to `inventory_tracking` scoped by `account_id`
- [ ] Given a dispatched `GamePlayEvent` with `partial: false`, when the GRE projector runs, then rows are written to `game_plays` and `life_change_tracking` ordered by `(occurred_at, sequence)`
- [ ] Given a dispatched `GamePlayEvent` with `partial: true`, when the GRE projector runs, then the row is written to `game_plays` with `partial = TRUE`
- [ ] Given any duplicate event (same idempotency key), when the projector runs a second time, then no duplicate rows are created
- [ ] Given an event with a sequence gap (`sequence > last_sequence + 1`), when the BFF processes it, then a structured warning is logged and a PostHog `daemon_event_gap_detected` event is emitted — the event is not discarded

---

### Wave 4 — Analytics Endpoints

**As a VaultMTG user with 20+ matches tracked,**
**I want** win rate, deck performance, and format stats to be visible in the app,
**so that** I can understand my performance trends.

**ACs:**
- [ ] Given a user with 20+ matches in `game_plays` (at least one per format), when `GET /api/v1/stats/deck-performance` is called, then win/loss/draw counts per deck are returned, scoped by `account_id`, excluding rows where `partial = TRUE`
- [ ] Given a user with collection data, when `GET /api/v1/collection` is called, then the current card inventory is returned scoped by `account_id`
- [ ] Given all analytics endpoints, when called without a valid Clerk JWT, then HTTP 401 is returned
- [ ] Given all analytics endpoints, when called with a valid JWT for user A, then data for user B is never returned

---

### Live Draft Track (Waves 2–4)

**As a VaultMTG user who is drafting in Arena,**
**I want** a live view at `/draft/live` that shows the current pack with card grades,
**so that** I can make informed picks without switching windows constantly.

**ACs:**
- [ ] Given an authenticated user with a valid Clerk session, when they open `/draft/live` in a browser tab, then an SSE connection is established to `GET /api/v1/events`
- [ ] Given the daemon emits `draft.pack` events, when the SSE consumer receives them, then the current pack updates in the UI within 2 seconds
- [ ] Given the SSE connection drops, when the `useDraftEventStream` hook detects disconnection, then it attempts reconnect with exponential backoff
- [ ] Given a user navigates away from `/draft/live`, when the component unmounts, then the SSE connection is closed cleanly (no memory leak)
- [ ] Given a draft session ends (`draft.ended` event), when the state machine receives it, then the UI shows a "Draft complete" state

## 7. Known Risks and Week 2 Bailout Trigger

### Risk 1 — Parser format drift

If #1501 (spike) reveals that Arena's log format has drifted significantly from the existing testdata, parser extraction becomes a repair job rather than a refactor. If the spike estimate for repair exceeds 5 days, invoke the Week 2 bailout trigger.

### Risk 2 — GRE session correlation complexity

The GRE event correlation (ADR-012) involves buffering and partial flush logic that interacts with daemon shutdown and stale-buffer sweep. If ADR-012 implementation in #1508 and #1519 takes longer than expected, the GRE projector (#1512) and stats endpoints (#1513) will slip.

### Risk 3 — SSE auth complexity

#1387 is a spike because EventSource cannot send an `Authorization` header. If Clerk session cookies cannot be made to work for SSE authentication within the spike, the live draft track may require an alternative auth mechanism (short-lived token, WebSocket). This could delay #1388–#1391 by 3–5 days.

### Week 2 Bailout Trigger

**Condition**: If, at the end of Week 2, any of the following are not green, invoke the v0.3.0-lite scope reduction:
- GRE projector (#1512) integration tests are not passing
- Collection delta projector (#1511) integration tests are not passing
- Quest projector (#1510 quest branch) integration tests are not passing

**v0.3.0-lite scope**: Drop game-play projector (#1512), collection delta projector (#1511), quest projector (#1510 quest branch), and their dependent analytics endpoints (#1514 rank progression, result breakdown, collection). Defer to v0.3.1.

**v0.3.0-lite deliverables** (match history + win rates + draft analytics only):
- Match history data flows from daemon through game_plays (match result only, no life total history)
- DeckPerformance and WinRateTrend endpoints (#1513) from match-only data
- DraftAnalytics endpoint (#1514 draft branch)
- Live draft overlay (#1390)
- Settings page (#1515)
- CSP headers (#1517)

v0.3.0-lite still satisfies the alpha invite gate: testers can validate match data accuracy, win rates, and the live draft view. Full telemetry parity is completed in v0.3.1.

**Decision deadline**: End of Week 2 (approximately 2026-05-21). Ray makes the call.
