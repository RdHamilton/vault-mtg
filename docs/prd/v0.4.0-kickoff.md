# v0.4.0 Kickoff: "Closed Beta"

**Date**: 2026-05-08
**Theme**: Close the gap between telemetry parity and a shippable closed beta.
**Author**: Product Manager
**Milestone**: v0.4.0 — Board #30
**Beta target**: Closed beta opens August 18, 2026. Internal stretch goal: June 26, 2026.

---

## 1. Purpose

v0.4.0 delivers the closed beta. When this milestone closes, vetted waitlist invitees can sign up, install the daemon, connect their Arena account, and use the full draft and match-history product — with analytics endpoints live, settings configurable, and the site hardened for public traffic.

v0.3.0 completed all telemetry pipeline work (22 tickets). v0.4.0 builds on top of it with the remaining analytics surface, the user-facing settings page, infrastructure hardening, daemon stability improvements, and the beta invite flow.

## 4. User Stories (Wave 4)

### Analytics Endpoints

1. As a beta tester, I want to see my deck performance over time so that I can identify which decks are improving.
   **ACs:**
   - [ ] GET /api/v1/analytics/deck-performance returns win rates grouped by deck, date range, and format
   - [ ] GET /api/v1/analytics/win-rate-trend returns time-series win rate with configurable granularity
   - [ ] GET /api/v1/analytics/format-distribution returns a breakdown of games played by format
   - [ ] All endpoints are paginated per the standard defined in #1516
   - [ ] All endpoints are scoped by authenticated account_id
   - [ ] PostHog event fired on first analytics page load: analytics_viewed

2. As a beta tester, I want to see my draft analytics (pick grades, trophy rates) so that I can improve my draft decisions.
   **ACs:**
   - [ ] GET /api/v1/analytics/draft returns per-draft grade averages and win rates
   - [ ] GET /api/v1/analytics/rank-progression returns rank over time
   - [ ] GET /api/v1/analytics/result-breakdown returns win/loss/draw counts by format
   - [ ] GET /api/v1/analytics/collection returns collection size over time

### Settings Page

3. As a beta tester, I want a Settings page so that I can manage my account and daemon configuration.
   **ACs:**
   - [ ] /settings route exists and is protected (ProtectedRoute)
   - [ ] Page shows: display name, email (read-only from Clerk), connected daemon status
   - [ ] User can disconnect their daemon from the settings page
   - [ ] PostHog event fired on settings page load: settings_viewed

### CloudFront Security Headers

4. As the operator, I want CSP and security headers on CloudFront so that the SPA passes a basic security scan.
   **ACs:**
   - [ ] Content-Security-Policy header set to restrict script-src, connect-src, img-src
   - [ ] X-Frame-Options: DENY
   - [ ] X-Content-Type-Options: nosniff
   - [ ] Referrer-Policy: strict-origin-when-cross-origin
   - [ ] Headers applied to all CloudFront responses (not just HTML)

### Crisp Widget

5. As a beta tester, I want in-app chat so that I can get help without leaving VaultMTG.
   **ACs:**
   - [ ] Crisp widget loads in the SPA for authenticated users
   - [ ] Crisp user identity is set from Clerk user data (email, name)
   - [ ] setup_idle_90s event fires if user is authenticated and has been idle for 90s without completing setup
